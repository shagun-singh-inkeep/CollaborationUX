# Current state: collaboration-experience surfaces in Open Knowledge

All paths relative to `public/open-knowledge/` in the engine repo unless noted. Confidence labels: CONFIRMED (read in source), SUPPORTED (inferred from adjacent source/tests), NOT FOUND (negative search documented).

## 1. Presence & awareness (who's here)

- **CONFIRMED — Human presence is per-doc Yjs awareness.** `packages/core/src/types/awareness.ts:1-60` — `AwarenessUser { name, color, type: 'human', icon?, coeditor?, tabId, principalId? }` + `AwarenessState { user, mode: 'wysiwyg'|'source'|'idle'|'editing', cursor? }`. Human cursors rendered by `@tiptap/extension-collaboration-cursor` (`packages/app/src/presence/participant-model.ts:4-8`).
- **CONFIRMED — Agents never publish per-doc awareness.** `awareness.ts:4-9`: "Agents no longer publish per-doc awareness — their presence lives on the `__system__` Y.Doc's `agentPresence` map (precedent #3)." Two map-valued fields on `__system__` awareness: `agentFocus` (per-write: doc + `writeKind: 'write'|'edit'|'undo'|'rollback-apply'`, `awareness.ts:67-76`) and `agentPresence` (sustained: displayName, brand icon, color, currentDoc, `mode: 'idle'|'writing'`, ts; 5s TTL; cleared deterministically on MCP keepalive WS close) (`awareness.ts:89-111`). Server broadcasters: `packages/server/src/agent-presence.ts`, `packages/server/src/agent-focus.ts:1-50`.
- **CONFIRMED — PresenceBar renders both populations.** `packages/app/src/presence/PresenceBar.tsx` — avatar bar with `M_CURRENT_PRIMARY = 4` current-doc + `K_CROSSDOC_PRIMARY = 3` cross-doc primary avatars, overflow popover (lines 36-42). Agents get brand icons + brand colors; humans get initials (git-config principal) or a Lucide animal icon (synthesized "Curious Turtle"-style name) — gating rule at PresenceBar.tsx `shouldRenderAnimalIcon` docblock (~line 60-80). Multi-tab humans dedupe by `principalId` with "Name · N tabs" tooltip (`participant-model.ts:88-119`).
- **CONFIRMED — agent `'writing'` vs human `'editing'` are deliberately distinct mode tokens** so CSS/tests can distinguish agent live-writes from human cursor activity on the same avatar attribute (`awareness.ts:98-108`).

## 2. Identity (who anyone *is*)

- **CONFIRMED — Human identity = per-project "principal".** `packages/server/src/principal.ts:38-104` — `.ok/local/principal.json` holds `{ id: principal-<UUID>, display_name, display_email, source: 'git-config'|'synthesized', created_at }`. Display fields refresh from `git config user.name/email` each load; id immutable. One principal per project root — there is no multi-user registry.
- **CONFIRMED — Browser fallback identity is a random pseudonym.** `packages/core/src/utils/identity.ts:229-315` — "Adjective Animal" name + pastel color persisted in localStorage (`ok-user-name-v3`); per-tab UUID. Used when no git-config principal resolves.
- **CONFIRMED — `principalId` on awareness is client-published and loopback-trusted.** `awareness.ts:20-27`: "the server is loopback-only and does not validate this field. If non-loopback connections ever ship, the publication site must switch to server-authoritative attribution."
- **CONFIRMED — Agent identity = per-MCP-session.** `packages/server/src/mcp/agent-identity.ts:15-25` — `AgentIdentity { connectionId (UUID per session), clientInfo { name, version }, displayName, colorSeed }`. Brand mapping (`claude-code`→Claude, `cursor-vscode`→Cursor, `codex-mcp-client`→Codex, `local-agent-mode-*`→Claude, unknown→'bot'/'Agent') in `packages/core/src/utils/identity.ts:133-185`. `clientInfo.name` is sanitized (control chars, 128-char cap) at `agent-identity.ts:60-70` because it is attacker-controllable input.
- **CONFIRMED — No auth on the collab transport.** `packages/server/src/auth-token-schema.ts:1-30`: all token fields optional; "Legacy clients... that never set a token must continue to authenticate cleanly"; token carries `principalId` but it is not verified. Defense is network-topological: the server refuses non-loopback peers for host-shape endpoints (`packages/server/src/loopback.ts:1-55`, incl. DNS-rebinding Host-header check).

## 3. Attribution of edits (who changed what)

- **CONFIRMED — Fixed 5-way writer taxonomy** (precedent #25): `agent-<id>`, `principal-<uuid>`, `file-system`, `git-upstream`, `openknowledge-service` (engine-repo root spec `specs/2026-07-06-timeline-upstream-author-attribution/SPEC.md` §8; `packages/core/src/shadow-repo-layout.ts`).
- **CONFIRMED — Every mutating HTTP handler threads actor identity at entry** (`extractAgentIdentity` for agent writes; `extractActorIdentity(body, getPrincipal)` for rename/rollback — body `principalId` is ignored, server's own principal is the only source). Enforced by meta-test `packages/app/tests/integration/attribution-sweep-coverage.test.ts`. Origin objects are per-session frozen (`session.origin`, precedent #24) so CRDT transactions carry attributable origins and per-session undo works via `Y.UndoManager({ trackedOrigins })`.
- **CONFIRMED — Contributor accumulator → git shadow commit bodies.** `packages/server/src/contributor-tracker.ts:33-139` — per-writer entry `{ writerId, displayName, colorSeed, docs, actor { principalId, agentType, clientName, clientVersion, label }, summaries[], previousPaths[] }`, drained into `ok-contributors: {json}` lines on WIP commits in the shadow repo (`formatContributorsFrom`, lines 209-231). So "who + which docs + optional per-write change-notes" is durable in git history.
- **CONFIRMED — Agent write summaries.** `packages/server/src/agent-write-summary.ts` — optional per-tool-call `summary` (80-char cap, line-terminator sanitized against commit-body injection, whitespace-only = absent). Rendered as collapsible bullets under the author line in the Timeline (`TimelinePanel.tsx:262-355`).
- **CONFIRMED — Live agent-change indicator: write-flash.** `Y.Map('agent-flash')` side-channel (`awareness.ts:113-119`) drives a CSS line-flash in both editors — CodeMirror `packages/app/src/editor/plugins/agent-flash-source.ts:1-30`, TipTap equivalent in `TiptapEditor.tsx`. Entries auto-evict after 30s.
- **CONFIRMED — Agent Activity Panel (per-agent session view).** `packages/app/src/lib/use-activity-panel.ts:1-70` — fetches `GET /api/agent-activity?agentId=`, subscribes to CC1 `session-activity` + `agentPresence` for a per-file "writing…" indicator; per-burst lazy diffs (`ActivityPanelBurstRow.tsx`, `ActivityPanelDiffView.tsx`, `ActivityPanelFileRow.tsx`). This is the "what did this agent do this session" surface, distinct from the Timeline.
- **CONFIRMED — Agent vs human origin is distinguishable end-to-end** in the data path: distinct transact origins per session, distinct writer-id prefixes, actor tuple in commit bodies, distinct presence channels, distinct icons in Timeline. What is NOT carried: any in-document per-character authorship mark (no Yjs-level per-item author metadata is surfaced; attribution granularity is per-write/per-commit, not per-span).

## 4. Timeline / history UI (what a user sees about past changes)

- **CONFIRMED — TimelinePanel** (`packages/app/src/components/TimelinePanel.tsx`, 825 lines): polls `GET /api/history` every 10s; reverse-chron rows of WIP/upstream entries (checkpoint rows filtered out, lines 129-134 — checkpoints are restore points, not user-facing history). Per row: relative time, author display name (`displayAuthor` lines 196-203: structured contributors first, `Auto-save` fallback), contributor icon (`ContributorIcon` lines 216-260: agent brand icons with brand colors; `HardDrive` for "File System"; `GitBranch` for upstream; `User` for humans/unknown), summary bullets, inline expandable diff vs live doc (lazy `react-diff-view`), and a per-row Restore button → confirm dialog → `POST /api/rollback`.
- **CONFIRMED — TimelineEntry shape:** `packages/core/src/types/timeline.ts` — `{ sha, timestamp, author, authorEmail, type: 'checkpoint'|'wip'|'upstream'|'park', message, contributors: ShadowContributor[], checkpoint }`.
- **CONFIRMED — Upstream (git pull) author recovery is implemented in server code.** `resolveUpstreamChanges` in `packages/server/src/server-factory.ts`, unit-tested at `packages/server/src/resolve-upstream-author.test.ts:8-14`: "maps each changed `.md`/`.mdx` doc to the newest non-merge commit author... so the reconcile can be attributed to a real author instead of the 'File System' writer." Corresponds to the engine-repo-root draft spec — implementation appears at least partially landed.
- **CONFIRMED — Agents see the same history.** MCP tools `history` (`packages/server/src/mcp/tools/history.ts:1-50` — entries with `version` SHA, author, authorEmail, kind), `restore_version`, `checkpoint`. Folder-level `FolderTimelineCard.tsx` also exists.

## 6. Sharing & who can currently join (and as whom)

- **CONFIRMED — Live transport is loopback-only.** `packages/server/src/loopback.ts` — host-shape endpoints refuse non-loopback peers; DNS-rebinding defense. The federated spec's current-state evidence notes `--host 0.0.0.0` is technically possible but nothing authenticates such a connection. Practical consequence: real-time multi-human collaboration across machines does not exist today; the "multiple humans" presence path is exercised by multiple tabs/browser profiles on one machine (`presence-multi-tab.test.ts`).
- **CONFIRMED — Sharing is asynchronous, via the GitHub substrate.** `ShareButton.tsx` + MCP `share_link` (`packages/server/src/mcp/tools/share-link.ts:1-52`): builds `https://openknowledge.ai/d/<base64url(github-blob-url)>` pinned to current branch; read-only against the working tree; freshness warnings when the target isn't pushed (`ShareFreshnessWarning.tsx`, `share/freshness.ts`). Recipient *receives a copy* into their own OK install (clone / open-local flow: `ShareReceiveDialog.tsx`, `CloneDialog.tsx`). Publishing to GitHub is an explicit human wizard (`PublishToGitHubDialog.tsx`); agents are deliberately not allowed to publish (`share-link.ts:10-15, 44`).
- **CONFIRMED — Human↔human collaboration = git push/pull.** Teammates' changes arrive via `git pull` → file-watcher reconcile → timeline entries (formerly "File System", now upstream-author-attributed, §4). Auto-sync (github-sync spec) pushes `Auto-save` commits every ~120s when enabled.
- **CONFIRMED — `ok cowork` is NOT a co-working/LAN feature.** `packages/cli/src/commands/cowork.ts:1-33` — hidden command that builds a `openknowledge.skill` bundle for Claude Desktop upload.

## 7. Permissions / roles / read-only

- **CONFIRMED (negative) — No permission, role, or view-only concept for humans.** Searched `readonly|read-only|view-only|viewer|permission|role` across server+app: hits are (a) TypeScript `readonly` keywords, (b) read-only *file viewers* for non-markdown assets (`TextViewer.tsx`, `ViewerStatusPane.tsx`), (c) `share_link`'s "read-only against the working tree" git semantics, (d) `github-permissions.ts` + `use-no-push-permission-toast.ts` — GitHub *repo* push-permission checks (access control is delegated to GitHub), (e) editor gating while disconnected/mounting. ## 7b. Authored-delta feasibility (code trace, 2026-07-08)

Can the Timeline show "what THIS actor changed" (`commit vs parent`) instead of today's `commit vs live` restore-preview?

- **CONFIRMED — clean parent chain.** Every WIP commit is `commit-tree treeSha -p parentSha` where parent = the writer's own prior ref tip (`packages/server/src/shadow-repo.ts:562-588`, `:429-459`). `park:` commits and auto-consolidation checkpoints are already excluded from the timeline (`timeline-query.ts:9-16`, `shadow-repo-layout.ts:629-638`) and don't rewrite the WIP parent chain. So `commit vs parent` is mechanically available.
- **CONFIRMED — `/api/history/:sha` returns doc content at a sha** via `git show <sha>:<historicalPath>` with rename-chain resolution (`api-extension.ts:8058-8081`). No parent-diff endpoint today; `TimelineEntry` carries only `sha`, no parent (`packages/core/src/types/timeline.ts:9-26`); diff is computed client-side `historical→live` (`use-timeline-entry-diff.ts:107-124`).
- **CONFIRMED — THE LIMIT: shared-tree drains.** One L2 drain builds a single whole-content-dir tree (`buildWipTree` stages the entire content root, `shadow-repo.ts:503-518`) and every writer active in that ~15s window gets a commit pointing at that SAME tree (`persistence.ts:843-906`). So `commit vs parent` for writer A = whole-dir-now − whole-dir-at-A's-prev-commit → conflates any co-window writer B. Engine comment states it: "per-writer commits in the same L2 drain share a tree SHA; tree-level diff cannot isolate one writer's contribution" (`agent-activity.ts:8-9`).
- **CONFIRMED — durable attribution = names only.** `ok-actor:`/`ok-contributors:` body lines record per-writer doc names + free-text summaries, no line/char ranges (`shadow-repo-layout.ts:507-524`, `contributor-tracker.ts:33-67`).
- **CONFIRMED — finer data exists but is ephemeral.** True per-actor per-doc before/after burst diffs are computed from each session's in-memory `Y.UndoManager.undoStack` (`agent-activity.ts:100-193`, `agent-sessions.ts:516`), GC'd on keepalive-WS close, live agent sessions only, never persisted — can't back historical rows.
- **VERDICT:** Moderate. TIER 1 (small): `parent vs commit` endpoint + parent sha in payload — fixes the common single-writer-per-window case (incl. the "hi"→"hey" example, which is sequential separate commits). TIER 2 (write-path change): true per-actor isolation needs durable burst diffs or per-writer-scoped WIP trees.

## 8. How agent edits arrive vs human edits (origin separability)

- **CONFIRMED — Write-surface table (engine CLAUDE.md "Write surfaces"):** W1 WYSIWYG (human, XmlFragment), W2 source (human, Y.Text), W3 agent HTTP API (`replaceRawBody` / `composeAndWriteRawBody` via MCP `write`/`edit` tools), W4 disk/file-watcher, W5 agent undo (`applyAgentUndo`), W6 property panel. Agent writes are batch HTTP writes ("agents don't edit, they batch-write" — `awareness.ts:103-105`), humans are keystroke-level CRDT edits. Origins, writer IDs, presence channels, and undo stacks are all partitioned per surface, so origin *is* distinguishable at every layer of the data path.
